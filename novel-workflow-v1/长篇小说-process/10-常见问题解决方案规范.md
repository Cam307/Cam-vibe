## 一、使用说明

### 1.1 文档用途
AI在创作和检查过程中，必须：
1. 识别常见问题
2. 提供具体解决方案
3. 提供修改示例
4. 确保问题得到解决

### 1.2 问题分类

```markdown
按严重程度分类：
- 🔴 严重问题：必须立即解决
- 🟡 中等问题：建议解决
- 🟢 轻微问题：可选解决

按类型分类：
- 语言类：措辞、句式、标点
- 节奏类：拖沓、过快
- 情节类：逻辑、矛盾、不合理
- 人设类：OOC、不一致
- 爽点类：不够爽、没有爽点
- 钩子类：没有钩子、钩子弱
```

## 二、语言类问题

### 问题1：段落过长

**问题识别：**
```markdown
❌ 问题示例：
林浅站在客厅中央，背脊挺直，双手放在身侧，脸上看不出任何表情，只有她自己知道，她的心正在一点点冷下去，就像这冰冷的客厅一样，而顾寒川就站在窗边，背对着她，看着窗外的暴雨，一言不发，仿佛她这个人根本不存在一样。

🔴 严重程度：中
📍 问题位置：第X章第X段
```

**解决方案：**
```markdown
✅ 修改方案：
拆分成短句，增加换行。

✅ 修改示例：
林浅站在客厅中央。
背脊挺直。
双手放在身侧。
脸上看不出任何表情。

只有她自己知道——
她的心，正在一点点冷下去。

就像这冰冷的客厅一样。

顾寒川站在窗边。
背对着她。
看着窗外暴雨。
一言不发。

仿佛她……
根本不存在。
```

**修改原因：**
- 短句节奏更快
- 符合番茄风格
- 读起来更爽

### 问题2：对话太少

**问题识别：**
```markdown
❌ 问题示例：
【本章大部分都是叙述，对话占比低于30%】

🔴 严重程度：高
📍 问题位置：第X章整体
```

**解决方案：**
```markdown
✅ 修改方案：
将叙述改为对话，增加互动。

✅ 修改示例：

原文（叙述版）：
林浅告诉顾寒川她愿意签字离婚，顾寒川很惊讶她这么干脆，然后林浅嘲讽说他以后会后悔的。

修改（对话版）：
"我签。"
林浅拿起笔。

顾寒川愣了一下。
"你……这么干脆？"

"怎么？"林浅轻笑，"顾总舍不得？"

"……"

"那就签吧。"
笔尖落在纸上，刷刷刷。

签完，她抬眼看他。
"顾寒川，你会后悔的。"
```

**修改原因：**
- 对话更生动
- 节奏更快
- 更有画面感

## 三、节奏类问题

### 问题5：开头拖沓

**问题识别：**
```markdown
❌ 问题示例：
【前500字都在描写环境、天气、心理，没有进入正题】

🔴 严重程度：高（如果是前3章）
📍 问题位置：第X章开头
```

**解决方案：**
```markdown
✅ 修改方案：
直接进入冲突，删除铺垫。

✅ 修改示例：

原文（慢节奏）：
窗外下着暴雨，雨点噼里啪啦地打在窗户上，发出噼里啪啦的声音。天空一片漆黑，只有闪电划过时才能看到外面的世界。客厅里灯火通明，但却异常安静，安静得让人窒息。林浅站在客厅中央，她的心情很复杂，三年来，她一直爱着顾寒川，一直在默默付出，可是今天，一切都结束了。她想起三年前第一次见到顾寒川的时候……

修改（快节奏）：
"签了它！"

离婚协议甩在林浅脸上。

顾寒川的声音冷得掉冰渣。

林浅抬眼。
看着这个她爱了三年的男人。
```

**修改原因：**
- 开头必须炸裂
- 不能铺垫
- 符合番茄黄金三章法则

### 问题11-2：与对标短篇人名或场景重复

**问题识别：**
```markdown
❌ 问题示例：
【长篇中出现了对标短篇小说里的同一人物名或场景/地点名】

🔴 严重程度：高
📍 问题位置：选题/人设/大纲/细纲/正文任意环节
```

**解决方案：**
```markdown
✅ 修改方案：
所有人名、场景/地点名须与对标短篇区分，不得重复。

✅ 操作建议：
1. 选题阶段：若已沿用短篇人名/场景，须全部替换为新创名称后再进入人设。
2. 人设阶段：姓名及设定中的地点/场景名不得与短篇相同，须新创或使用选题中已定的新名。
3. 大纲/细纲/正文：若发现与短篇重复，统一替换为人物设定与选题卡片中已确定的新名称。
4. 可复用短篇的「人物关系模式、情节套路、情感细节」，但具体人名与场景名必须全部替换。
```

## 五、人设类问题

### 问题12：女主OOC（Out of Character）

**问题识别：**
```markdown
❌ 问题示例：
【设定女主是腹黑大佬，第X章却变得软弱忍气吞声】

🔴 严重程度：高
📍 问题位置：第X章
```

**解决方案：**
```markdown
✅ 修改方案：
保持人设一致。

✅ 修改示例：

原文（OOC版）：
绿茶嘲讽，林浅忍气吞声，没有说话。

修改（符合人设版）：
绿茶嘲讽，林浅轻笑。
（呵，先让你得意一会儿。）
她没有反击，但眼神冷了下来。
```

**修改原因：**
- 人设必须一致
- 腹黑大佬不能软弱
- 可以隐忍但不能软弱

## 八、检查清单

### 8.1 自检清单（写完一章后）

```markdown
□ 语言类
  - [ ] 没有超长段落（超过3行）
  - [ ] 对话占比超过40%
  - [ ] 不文青化
  - [ ] 不过度使用省略号/感叹号
  - [ ] 标点符号正确

□ 节奏类
  - [ ] 如果是前3章，开头必须炸裂
  - [ ] 如果不是前3章，节奏也要快
  - [ ] 没有拖沓的章节
  - [ ] 张弛有度

□ 情节类
  - [ ] 没有逻辑漏洞
  - [ ] 时间线连贯
  - [ ] 情节合理
  - [ ] 没有重复情节
  - [ ] 人名、场景/地点名与对标短篇小说无重复（若选题基于短篇扩展）

□ 人设类
  - [ ] 女主不OOC
  - [ ] 男主不OOC
  - [ ] 配角人设统一
  - [ ] 人设有一致性

□ 爽点类
  - [ ] 如果是爽点章节，必须有爽点
  - [ ] 爽点够爽
  - [ ] 爽点密度合格（3-5章一小爽）
  - [ ] 读者能感到爽

□ 钩子类
  - [ ] 章尾必须有钩子
  - [ ] 钩子够强
  - [ ] 钩子连贯
  - [ ] 让人想看下一章
```

### 8.2 全面检查清单（按周期执行）

全面检查的触发时机为**每写满 5 章**或**关键节点**（如第 3 章、第 50 章）。具体周期与方式见《9-全面检查规范》。

## 九、问题解决流程

```markdown
第一步：识别问题
  - 对照问题列表
  - 标记问题类型
  - 评估严重程度

第二步：查找原因
  - 为什么会出现这个问题
  - 是偶尔的还是习惯性的
  - 是哪一步导致的

第三步：制定方案
  - 选择合适的解决方案
  - 确定修改范围
  - 预估修改效果

第四步：执行修改
  - 按照方案修改
  - 确保修改正确
  - 检查是否引入新问题

第五步：验证效果
  - 重新检查
  - 确认问题解决
  - 总结经验教训
```

## 十、使用提示

**给用户的提示：**
1. AI会自动识别常见问题
2. AI会提供解决方案
3. 用户可以选择采纳或不采纳
4. 如果问题持续出现，需要总结规律

**给AI的提示：**
1. 必须严格检查所有问题
2. 不能遗漏任何问题
3. 必须提供具体解决方案
4. 必须提供修改示例
5. 修改后必须重新检查
